Bluetooth Earphone Companion App

Design Choices and Process Report (CENG + ID, Spring 2026)

1. Concept Description

Product Purpose

The product is a Bluetooth earphone companion app that helps users connect, monitor, and personalize earphones with minimal friction across daily listening scenarios.

User Needs

  • Quick and understandable Bluetooth pairing and connection feedback.
  • At-a-glance battery/device state from a central home view.
  • Fast sound tuning through presets or simple manual EQ controls.
  • Immediate access to listening mode changes (NC/Normal/Transparency).
  • Simple account and settings flows for device and profile management.

Key Functions

  • Bluetooth pairing and success/failure state messaging.
  • Home dashboard with device status and quick actions.
  • Sound presets and bass/mid/treble custom sliders with save.
  • Noise mode switching and device settings configuration.
  • Authentication (login/register/forgot password) and profile overview.

2. Information Architecture and Main Flow

We merged paper wireframes, flow sheets, and Figma v3 into one canonical flow:

  • Launch/Splash -> Home.
  • Home -> Add Device -> Bluetooth Pairing -> Success -> Home.
  • Home -> Sound/Volume -> Presets -> Custom sliders -> Save.
  • Home -> Noise mode screen.
  • Home -> Settings -> Device Detail -> Automatic Power-Off.
  • Home -> Profile (auth flows + post-login pages).

Primary references: docs/page-flows.md and docs/wireframes/WIREFRAME-ANALYSIS.md.

5. Reflection

Choices and Alternatives

  • Home was chosen as a central operational hub to reduce navigation overhead.
  • Preset-first sound control was kept for speed; custom sliders are secondary.
  • Alternative considered: merge noise controls fully into Sound. Rejected for discoverability reasons.

Challenges

  • Different tab logic across early paper and later digital assets.
  • Potential duplication of intent between similar screens in multiple artifacts.
  • Need for clear state mapping in Bluetooth flow (empty/searching/success/connected).

User Considerations

  • Frequent tasks should be reachable in 1-2 taps.
  • First-time users need direct and visible add-device onboarding.
  • Status feedback should always be explicit and consistent.

6. Conclusion

The resulting UI architecture aligns connection tasks, sound personalization, and profile/settings management in one coherent flow suitable for iterative refinement in high-fidelity stages.
